# 要件定義書：コマンドプロンプトリファレンス

## 1. 背景 / 目的

Windows の **コマンドプロンプト（cmd.exe）** と **バッチファイル（.bat/.cmd）** を対象に、コマンドごとの解説・使用例・注意点、さらに文法/トピックを体系立ててまとめる。

### 解決したい課題

- 情報が散らばっていて「知りたいことに最短で到達しにくい」
- 例が環境依存で、そのまま動かない/危険なケースがある
- cmd.exe と PowerShell が混同されがちで、動作差が理解しづらい

### 成功の定義（KPIの代わり）

- 1ページのテンプレが確立し、同じ型で増やせる
- 「コマンド名」で引けば目的のページに到達できる（索引/検索）
- 例が **コピペで再現** でき、落とし穴が先回りで説明されている

## 2. 対象ユーザー

- 初心者：cmd.exe の基本操作を知りたい
- 中級者：バッチでの実装パターン、落とし穴、互換性を知りたい
- 参照用途：オプションや構文をすぐ確認したい

## 3. スコープ

### 3.1 対象（In scope）

- コマンドプロンプト（cmd.exe）の主要コマンド解説
- コマンドの **使用例（最小/実用/オプション付き）**
- バッチファイル文法（変数展開、if/for、リダイレクト、関数っぽい書き方 等）
- 横断トピック（文字コード、エスケープ、エラー処理、遅延展開、パス 等）

### 3.2 対象外（Out of scope：初期）

- PowerShell の体系的解説（差分説明は必要に応じて扱う）
- Linux/macOS シェルの体系的解説
- GUI操作やWindows設定の一般解説（必要なら topic で最小限）

## 4. コンテンツ要件（ページ設計）

### 4.1 コマンドページ（例：`docs/commands/copy.md`）

- **概要**（何をするか）
- **代表的な使い方**（最小/実用）
- **オプションを使用した例**（最低1つ）
- **オプション/引数**（表）
- **よくある落とし穴**（バッチと対話差、クォート、権限、文字コード 等）
- **関連リンク**（内部リンク中心）

テンプレ：`docs/TEMPLATE_COMMAND.md`

### 4.2 バッチ文法ページ（例：`docs/batch/variables.md`）

- 概要
- 基本
- 実用例（`setlocal`/`endlocal` 含むのが基本）
- 注意点（遅延展開、cmd.exe限定/バッチ限定、互換性）
- 関連

テンプレ：`docs/TEMPLATE_BATCH.md`

### 4.3 横断トピック（例：`docs/topics/encoding.md`）

- 概要
- よく出る症状 → 原因 → 対処（トラブルシュート型）
- 関連ページ（コマンド/文法への導線）

## 5. ナビゲーション / 情報設計

### 必須

- カテゴリ分け（commands / batch / topics）
- コマンド一覧（A-Z / あいうえお的でも可）
- 各ページに「関連」セクション（回遊性）
- タグ（例：`encoding` / `filesystem` / `network` / `admin`）
- サイト内検索（全文検索：コマンド名/オプション/構文でヒット）

### 検索要件（詳細）

- **コマンド検索**：コマンド名（完全一致/前方一致）を強く優先し、本文一致も拾う
- **アルファベット索引**：A-Zで辿れる一覧（`a` には `assoc` / `attrib` のようにまとまる）
- **逆引き（使用用途）**：
  - 「やりたいこと」からコマンドへ辿れる（例：ファイル操作/ネットワーク/環境変数/プロセス/権限/文字コード）
  - タグやカテゴリで **絞り込み** ができる
- **別名/同義語**：よくある呼び方（例：`環境変数` / `変数`、`コピー` / `複製`）でもヒットする

### 検索の実装方針（非DB）

- **DBなし**で実現する（ビルド時に索引を生成し、静的に配信）
- 検索は「入力→即時ヒット」が体感できる速度を目標とする
- 検索/索引/逆引きのため、記事には **frontmatter（メタデータ）を必須** とする

### 出力例（実行結果）の扱い

- 使用例には **原則として出力例（実行結果）を併記** する（結果が空/環境依存で無意味な場合は省略可）

## 6. 品質要件（コンテンツ品質）

- **例はコピペで動く**（前提条件がある場合は箇条書きで明記）
- 破壊的操作（削除/上書き/レジストリ等）は **危険表示** と **dry-run** を先に出す
- Windows差分（10/11、権限、文字コード）は必要に応じて明示（「動作環境」欄は **必要なページのみ**）
- 外部リンクは最小限（貼るなら要点を一言添える）

規約：`docs/CODING_STANDARDS.md`

## 7. 非機能要件（サイト/運用）

> MVPで「公開」まで含める（ホスティング/生成ツールの細部はTBD可）。

- **ホスティング**：静的ホスティング（例：GitHub Pages）で公開する
- **サイト生成**：TBD（例：MkDocs / Docusaurus / Astro / 素のMarkdown）
- **URL設計**：TBD（英小文字、拡張子あり/なし）
- **更新運用**：PRベース or 直コミット（個人運用なら後者でも可）
- **デプロイ計画**：`docs/DEPLOYMENT_PLAN.md` に定義する（MVPで本番公開まで実施）

## 7.1 技術スタック選定（要件）

### 採用技術一覧（この段階）

| 区分 | 採用 | 理由/備考 |
| --- | --- | --- |
| 言語 | TypeScript | JSを使うならTSを使用する方針 |
| コンテンツ | Markdown + MDX（必要なページのみ） | 記事作成のしやすさ。拡張が必要な箇所だけMDX |
| サイト生成 | Astro | TypeScript前提の静的サイト生成 |
| UI（Material公式） | `material-web` | Material Design公式のWeb Components |
| ホスティング | GitHub Pages | MVPで公開まで含める |
| 検索 | 静的全文検索（ビルド時索引） | **DBなし**で高品質検索を実現する |
| 検索の具体候補 | Pagefind（候補） | Astro等の静的サイトと相性が良い方式 |
| DB | なし | コメント/投稿/ログイン/お気に入り/分析が不要のため |
| バックエンド | なし | 静的サイトのみで運用する |

### 方針 / 制約

- **マイナーな言語、過度に古い言語、難読言語は避ける**
- バックエンド言語として **Haskell は却下**（採用しない）
- JavaScript系で実装する場合は **TypeScript を使用**する
- 記事作成は **Markdown が楽なら Markdown を使う**
- CSS/UIは **Material Design を用いる**
- UIライブラリは **Material Design公式のCSSライブラリ（公式）を使用**する

### 推奨スタック（静的サイト前提）

> 要件（Markdown、TS、Material、検索/タグ、公開）を満たしやすい「推奨案」。

- **サイト生成**：Astro（TypeScript）
- **コンテンツ**：Markdown（必要なページのみMDXも使用）
- **UI（Material公式）**：`material-web`（Material Web Components）
- **検索**：静的全文検索（ビルド時に索引生成。例：Pagefind 等）
- **ホスティング**：GitHub Pages

### バックエンド（現時点）

- **不要**（静的サイトのみで運用する）

### バックエンドが必要になった場合（想定）

> MVP時点では導入しないが、将来の拡張で必要になりうる機能と、その時の選定方針をここに定義する。

#### 想定する「必要になる機能」

- （将来もし必要になったら）コメント/投稿/お気に入り同期/管理画面 等

> 現時点の要件としては、コメント・ユーザー投稿・ログイン・お気に入り・分析は **不要** とする。

#### バックエンド導入時の制約（技術選定）

- **TypeScriptを使用**（マイナー/難読言語は避ける）
- **Haskellは採用しない**
- 可能なら **運用負荷が低い構成**（サーバ管理を最小化）

#### 推奨候補（優先順）

- **サーバレス（推奨）**：Cloudflare Workers / Pages Functions 等（TypeScript）
  - 小さなAPI（コメント送信、投稿受付）に向く
- **Node.js（TypeScript）API**：Express/Fastify 等
  - 機能が増えた場合の柔軟性を優先

#### データ保存（候補）

- まずは **外部サービス/マネージドDB** を優先（運用負荷を下げる）
- 自前DBを持つ場合は **PostgreSQL** を第一候補（一般的で情報が多い）

#### セキュリティ/運用上の最低要件

- スパム対策（レート制限、CAPTCHA等の検討）
- 入力検証、XSS対策（Markdown/HTMLの扱いを明確化）
- 管理者権限（承認/削除）の導入

## 8. 初期リリース（MVP）要件

### MVPに含めるもの

- `docs/` の構造と規約・テンプレ
- コマンドページを数本（例：`echo`、`set`、`cd`、`dir`、`copy`）
- バッチ文法ページを数本（例：変数、遅延展開、if、for）
- topics を1本（例：文字コード/文字化け）
- コマンド一覧（索引）ページ
- タグ運用（最低限：ページにタグを付けられる仕組み）
- サイト内検索（全文検索）
- 静的サイトとして公開できる状態（ビルド/配置手順を含む）

### MVPで後回しでもよいもの

- 自動テスト（リンク切れチェック等）
- 多言語対応

## 9. リスク / 留意点

- cmd.exe と PowerShell の混同により例が動かない問題（→ “cmd.exe限定” を明示）
- 文字コード問題（→ topic で早期に整理）
- 破壊的コマンドの誤用（→ dry-run と危険表示の標準化）

## 10. 用語

- cmd.exe：コマンドプロンプトの実体
- バッチ：`.bat`/`.cmd` ファイル
- 例：本文中のコピペ可能なコマンド/スクリプト断片

## 11. 決定事項（回答反映）

（反映済み）

- サイト内検索（全文検索）：**要る（MVPから）**
- タグ：**要る（MVPから）**
- コマンド一覧ページ（索引）：**要る（MVPから）**
- 各ページの「動作環境」欄：**必要な時だけ**
- 出力例（実行結果）：**基本すべて載せる**
- 日本語/英語の二言語対応：**要らない**
- 静的サイトとして公開：**MVPに含める**
