# nav-retract-1: 再現条件の整理

画面遷移時に「コマンド」の**追加サイドバー**（右に広がるパネル）が引っ込む **見た目のアニメ**（右→左）が期待どおり出ない件について、検証するときの条件を固定する。

## 期待値（仕様）

「期待通りに動かない」と書くときの **期待値** を次で定義する。ここが満たされない場合が不具合（バグ）として扱う。

### 共通（追加サイドバーの見え方）

- パネルはメインサイドバーの右に付く **縦長のパネル**である。
- **開く**とき: 左端を軸に、**右方向へ広がる**（横方向の伸び。CSS では主に `transform: scaleX` で表現）。
- **閉じる**とき: 開くときの逆で、**右から左へ畳んでいく**（横方向に細くなる）。

### A. マウスホバーで開く・離す（参照動作・既に満たしている想定）

| 操作 | 期待値 |
| --- | --- |
| コマンドにマウスを載せる | 追加サイドバーが **約 320ms** かけて開く（横に伸びるアニメ付き）。 |
| マウスを外してパネルが閉じる | 追加サイドバーが **約 320ms** かけて閉じる（横に縮むアニメ付き）。 |

※ ここは「正しい見本」。遷移時の期待も **視覚的にはこの「閉じる」と同種**（横に縮む）であること。

### B. 画面遷移（コマンド配下 → 別ページ）— 本バグの期待値

**前提:** コマンド記事など **URL が `/commands/` 配下**のページを表示している。  
**操作:** メインサイドバーで **コマンド以外**のリンク（HOME・検索・逆引き・バッチ・topics 等）をクリックし、別 URL へ遷移する。

| 条件 | 期待値 |
| --- | --- |
| **追加サイドバーが開いている**（コマンドにホバー中でパネルが見えている） | 1. クリックから **次のページが表示されるまでの間** に、追加サイドバーが **閉じる向きのアニメ**（横に縮む・右から左へ畳む）が **目視できる**こと。<br>2. アニメの時間感はホバー離脱時の閉じと **同程度（おおよそ数百 ms 以内）** であること（厳密な ms は実装と一致していればよい）。<br>3. その後、クリック先のページへ遷移すること。 |
| **追加サイドバーが閉じている**（コマンドにホバーしておらずパネルが見えない） | **目視できる「横に縮む」パネルは無い**のが正常。遷移だけが行われればよい（「パネルが無いのに縮むアニメ」は求めない）。 |

### 現状（不具合の言い方）

- **期待:** 上表 B の「開いている」行。  
- **問題:** パネルが開いている状態で遷移しても、**閉じるアニメが目に見えない**／**一瞬で消える**など、**A のホバー離脱時ほど明確な横方向の引っ込みが見えない** という報告がある。  
- コード上は遷移前にアニメ用のクラスや `keyframes` を掛けているが、**期待 B を満たしていない** = 「実装しているが効いていない／見えていない」可能性として扱う。

## 用語

| 用語 | 意味 |
| --- | --- |
| メインサイドバー | 左列のアイコン列（検索・HOME・コマンド…） |
| 追加サイドバー | 「コマンド」ホバー時に右に出るパネル（コマンド一覧・A/B…） |
| 遷移トリガー | コマンド以外の **メインサイドバーの `a.link`** をクリックして別 URL へ行く操作 |

## 前提（コード上の条件）

遷移用の処理（`setupCommandsPageNavLeave`）が動くのは次のときだけである。

1. **URL** の `pathname` に `/commands/` が含まれる（例: `/commanddictionary/commands/echo/`）。  
2. クリック対象が **`a.link`** かつ **コマンドのメインリンクではない**（HOME・検索・逆引き・バッチ・topics など）。

※ `base` が `/commanddictionary` のときは pathname もそれに合わせて解釈する。

## 推奨の手順（R1）

1. コマンド記事ページを開く（例: `/commands/echo/`、base 付きなら `/commanddictionary/commands/echo/`）。  
2. 「コマンド」アイコン上にマウスを置き、追加サイドバーが **完全に開いた**ことを確認する。  
3. マウスを **HOME（または検索など）に移動**し、**クリック**する（コマンドの `li` からは外れていることが多い）。  
4. 遷移直前の数フレームで、追加サイドバーが **右から左に畳まれる**か目視する。

## nav-retract-2: 調査結果（要約）

コードレビューと仕様比較に基づく。**ブラウザの DevTools での実測は手元環境に依存するため、必要なら別途記入する。**

| 論点 | 結論 |
| --- | --- |
| `commandSubmenuNavLeaveWidth`（`width: var(...) → 0`） | **`width` と `var()` の組み合わせは、環境によってキーフレーム補間が効かない／`animationend` が発火しない**ことがある（CSS の制約・実装差）。その場合、**450ms の `setTimeout` だけが効き**、画面は遷移するが **縮む見え方がほぼ無い**ように見える。 |
| `animationend` | アニメが実際に走れば `commandSubmenuNavLeaveWidth` 名で届く想定だったが、上記の通りアニメが無効に近いと **イベントも来ない**。 |
| ホバー離脱との差 | ホバー閉じは **`commandSubmenuRetract`（`transform: scaleX(1)→scaleX(0)`）** で安定して見えている。 |

### 対応（nav-retract-3 と一体で実施）

遷移時も **`commandSubmenuRetract` に統一**し、`command-submenu-nav-leave` 付与時はホバー用 `command-submenu-retracting` と同じキーフレームを使う。`animationend` の判定も `commandSubmenuRetract` に合わせる。実装は `src/components/SidebarNav.astro` を参照。

### nav-retract-4: 遷移タイミング

- **主経路**: `commandSubmenuRetract` の `animationend` → リスナ解除・タイマー解除のうえで `location` 遷移。
- **保険**: 320ms + 130ms 経過しても `animationend` が来ない場合に `console.warn` を出してから遷移（ユーザーが遷移で詰まらないようにする）。本番でもフォールバック発火時はコンソールに残るので、稀な環境差の調査に使える。

### 遷移の順序（実装の意図）と「先に遷移している？」への答え

意図している順序は次のとおりである。

1. メインの別 `a.link` のクリックを **`preventDefault`** し、**ブラウザ既定の即時遷移を止める**。
2. 追加サイドバーに `command-submenu-nav-leave` を付け、**約 320ms の引っ込み**（`commandSubmenuRetract`）を再生する。
3. **`animationend`**（失敗時はフォールバックタイマー）のあとで **`window.location.href = href`** する。

したがって、**正常系では「引っ込みのあとに画面遷移」**であり、**意図的に先に `location` だけ飛ばしているわけではない**。  
それでも「一瞬で消えてすぐ別ページ」に見える場合は、(a) `preventDefault` が効いていない、(b) 引っ込みが**見えていない**（下記の visibility の穴）、(c) DevTools で `animationend` が来ずフォールバックだけが効いている、などを疑う。

### visibility の穴（一瞬で見えなくなる原因になりうる）

`command-submenu-open` などを外した直後、**ベース** `.submenu` の `visibility: hidden` が一瞬だけ当たる。インラインで `visibility` を固定する**前**に `offsetWidth` 等で reflow すると、パネルが非表示のままキーフレームが走り、**横に縮むのが目に見えない**ことがある。対策として、`open` 系クラスを外す**前**に `submenu` に `visibility: visible` と `pointer-events: none` をインラインで付与してからクラスを外す（`SidebarNav.astro` の `setupCommandsPageNavLeave`）。

### 追加で疑う原因（再発時）

1. **`submenuLooksOpenForNavLeave` の誤判定（閉じ扱い）**  
   `getComputedStyle(visibility)` だけを見て閉じと判定すると、**まだ `transform` で畳まれている途中**なのに `visibility` だけ先に `hidden` 寄りに見えるタイミングがあり、**引っ込み遷移ルートに入らない**。判定は **`getBoundingClientRect()`（変形後の幅）と `scaleX` を優先**し、物理的に幅・スケールがゼロに近いときだけ閉じとする。

2. **同一 `@keyframes` の再適用**  
   直前に `command-submenu-retracting` で既に `commandSubmenuRetract` が走っていると、**同じ名前のアニメを付け直すだけでは再生されない**ことがある。遷移用クラスを付ける直前に **`animation: none` → reflow → `animation` 解除**してから付与する。
