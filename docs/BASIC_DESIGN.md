# 基本設計書：コマンドプロンプトリファレンス

## 1. 目的 / 前提

本書は、要件定義（`docs/REQUIREMENTS.md`）に基づき、静的サイトとして実装するための基本設計をまとめる。

### 前提（確定）

- 静的サイト（バックエンド/DBなし）
- Astro + TypeScript
- コンテンツは Markdown を基本、必要なページのみ MDX
- UI は Material Design（公式）：`material-web`
- 検索は **ビルド時に索引生成**して静的配信（例：Pagefind 等）
- デプロイは GitHub Pages（計画：`docs/DEPLOYMENT_PLAN.md`）

## 2. 全体構成（アーキテクチャ）

### 2.1 コンポーネント

- **サイト本体**：Astro（静的生成）
- **コンテンツ**：`docs/` 配下の Markdown/MDX（※最終的な配置は実装時に `src/content` 等へ移す可能性あり）
- **検索インデックス**：ビルド時生成（静的ファイルとして同梱）

### 2.2 データの持ち方（DBなし）

- 記事や索引情報は **ファイル（Markdown/JSON/TS）** で管理する
- タグ/逆引きは「ページのメタデータ（frontmatter等）」を元にビルド時に集約する

## 3. 画面設計（主要ページ）

### 3.1 トップ

- サイトの目的
- 大きい検索窓（コマンド名/用途/オプションで検索）
- 主要導線（コマンド一覧、逆引き、バッチ文法、topics）

### 3.2 検索ページ

- 入力フィールド
- 結果一覧（ヒット理由：タイトル/見出し/本文一致）
- 絞り込み（タグ/カテゴリ）

### 3.3 コマンド一覧（A-Z）

- A〜Zの見出し
- 各文字ごとのコマンドリンク（別名がある場合は参照も表示）

### 3.4 逆引き（使用用途）

- 目的カテゴリ一覧（例：ファイル操作、ネットワーク、環境変数、プロセス、権限、文字コード、ディスク等）
- カテゴリを選ぶと、該当コマンド/記事へ導線

### 3.5 記事ページ（コマンド/バッチ文法/topics）

- テンプレに沿った構成（`docs/TEMPLATE_COMMAND.md` 等）
- 「関連」導線（同カテゴリ、同タグ、関連topics）

## 4. ルーティング設計（案）

> Astro 側のディレクトリとURLは、実装時に確定する。

- `/`：トップ
- `/commands/`：コマンド一覧（A-Z）
- `/commands/<command>/`：コマンド記事
- `/batch/`：バッチ文法一覧
- `/batch/<topic>/`：バッチ文法記事
- `/topics/`：横断トピック一覧
- `/topics/<topic>/`：横断トピック記事
- `/search/`：検索
- `/reverse/`：逆引き（使用用途）

## 4.1 URL形式（確定）

- **pretty URL（拡張子なし）**：`/search/` のように末尾スラッシュで統一する
- 静的生成物は `dist/search/index.html` のように出力される前提（Astroの一般的な出力）
- GitHub Pages は **プロジェクトサイト配信**を基本とする  
  - 想定URL：`https://<owner>.github.io/commanddictionary/`
  - Astroの `base`：`/commanddictionary`（設定：`astro.config.mjs`）
  - ユーザーサイト（`https://<owner>.github.io/`）に切り替える場合は `base: "/"` に変更する
- 内部リンク / form `action` / クライアント側遷移先は `import.meta.env.BASE_URL` を考慮して生成する（実装では `withBase()` を使用）
- クエリ文字列を使う画面は次を標準とする  
  - 検索：`/search/?q=<query>&tag=<tag>`  
  - 逆引き：`/reverse/?tag=<tag>`
- URLエンコード規約  
  - `q` / `tag` は `encodeURIComponent` 相当でエンコードして埋め込む
  - 直接入力の値をそのまま連結しない

## 5. コンテンツ設計

### 5.1 ページ種別

- **command**：1コマンド1ページ（例：`copy`）
- **batch**：バッチ文法（例：遅延展開）
- **topic**：横断トピック（例：文字コード）

### 5.2 メタデータ（frontmatter等）の設計（案）

> DBなしで「索引/逆引き/タグ/別名」を実現するための最小メタデータ。**全ページ必須** とする。

- `title`：表示タイトル（例：`copy`）
- `type`：`command` / `batch` / `topic`
- `tags`：用途タグ（例：`filesystem`、`network`、`encoding`）
- `aliases`：別名/同義語（例：`コピー`、`複製`）
- `summary`：一覧や検索結果での短い説明（1〜2行）

### 5.3 逆引きカテゴリ（案）

タグは「記事に付ける最小単位」、逆引きは「人間が辿る目的カテゴリ」。

- 例：`filesystem` / `network` / `env` / `process` / `security` / `encoding` / `disk` / `text`

## 6. 検索設計

### 6.1 検索対象

- ページタイトル
- 見出し
- 本文
- `aliases`（別名）

### 6.2 検索の体験（要件）

- コマンド名の完全一致/前方一致を優先できること
- タグ/カテゴリで絞り込みできること
- 逆引き（使用用途）への導線があること

### 6.3 実装方針（案）

- Pagefind等の「静的全文検索」を採用
- 追加で、タイトル/別名は **別レイヤー（辞書）** として先頭ヒットさせることを検討

## 7. UI/デザイン

### 7.1 デザイン方針

- Material Design（公式コンポーネント）に寄せる
- 検索を最優先（最短操作）
- モバイル/PCどちらでも読めるレイアウト

### 7.2 主要UI部品（例）

- App bar（ヘッダ）
- 検索ボックス
- ナビゲーション（サイド/上部）
- 記事内のコードブロック（コピーしやすい）

## 8. 非機能設計

- **アクセシビリティ**：キーボード操作、コントラスト、見出し構造
- **パフォーマンス**：検索インデックスのサイズ、初回表示速度
- **セキュリティ**：外部入力を扱わない（静的）。MDXは使用範囲を限定
- **運用**：Markdown編集 → ビルド → GitHub Pagesへ自動反映

## 9. 未決事項（これ要りますか？）

以下は「基本設計書に入れる/確定する」必要があるか、判断が分かれやすい項目。
要るなら、この文書に具体仕様まで落とし込む。

- タグの標準セット（最初に決め切るか、育てるか）
- 逆引きカテゴリの固定（カテゴリ一覧を固定するか、タグから自動生成するか）
- 記事の frontmatter（上記の項目を全ページ必須にするか）
- 検索の優先度制御（「コマンド名を最優先」等の具体ルール）
- ダークモード対応

## 10. 決定事項（回答反映）

- **URL形式**：基本設計で確定する
- **タグ方針**：固定せず、運用しながら育てる
- **逆引き（使用用途）**：タグから自動生成する（タグ=用途カテゴリ）
- **検索優先度ルール**：後で調整（実装で試す）
- **ダークモード**：不要
- **frontmatter**：全ページ必須（`title/type/tags/aliases/summary`）
